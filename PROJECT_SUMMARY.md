# Hospital Management System (HMS) - Project Summary

## Project Overview
A full-stack Hospital Management System built with FastAPI (Python) backend and React + TypeScript frontend. The system manages patients, doctors, appointments, and billing.

---

## Tech Stack

### Backend
- **Framework**: FastAPI (Python 3.x)
- **Database**: PostgreSQL with SQLAlchemy ORM
- **Migrations**: Alembic
- **Authentication**: JWT-based auth (simplified for demo)
- **CORS**: Enabled for frontend communication
- **API Documentation**: Auto-generated by FastAPI at `/docs`

### Frontend
- **Framework**: React 19.x + TypeScript
- **Build Tool**: Vite 8.x
- **Styling**: Tailwind CSS 3.x + Custom CSS
- **State Management**: React Hooks (useState, useEffect, custom hooks)
- **Form Handling**: React Hook Form + Zod validation
- **HTTP Client**: Axios
- **Icons**: Lucide React
- **Toast Notifications**: React Hot Toast
- **Routing**: React Router DOM v7

- id, `patient_id`, `doctor_id`, `appointment_time`, status, `tenant_id`, `created_by`, etc. Status values: scheduled, completed, cancelled (see `AppointmentStatus`).

### Billing
- id, `patient_id`, `appointment_id`, amount, status, `tenant_id`, `idempotency_key` (where applicable), etc. (see `app/models/billing.py`).

## Recent Major Changes

### Patients: Global Patient Architecture (May 2026)

- **`patients.tenant_id` is ALWAYS NULL** in PostgreSQL: patients are global records and NEVER assigned a tenant_id. Migration: **`a3b4c5d6e7f8_make_patient_tenant_optional`**; model: `Patient.tenant_id` → `Mapped[uuid.UUID | None]` (always null).
- **Tenant context is derived ONLY from appointments**: A patient belongs to a tenant only when they have an active `Appointment` with `Appointment.tenant_id` set. Patient access control uses `patient_has_active_appointment_in_tenant()` — NOT `patient.tenant_id`.
- **CRITICAL INVARIANT**: Never write to `patient.tenant_id`. The field exists for legacy compatibility but remains permanently NULL.

### Doctor: patient detail, timeline, and quick workflows (April 2026)

Independent doctors get a focused **patient detail** experience (`backend/frontend/src/pages/doctor/DoctorPatientDetailPage.tsx`):

- **Activity timeline**: Grouped by calendar day (TODAY / YESTERDAY / long date) with clear hierarchy; **newest time first** within a day, with a stable tie-break (appointment before bill at the same instant). **Calendar** and **Indian Rupee** icons distinguish visits vs bills.
- **Inline actions** (where RBAC allows): **Mark completed** / **Cancel** for scheduled (or client “pending”) appointments via `PUT /api/v1/appointments/{id}`; **Mark as paid** for unpaid bills via `POST /api/v1/bills/{id}/pay` (`appointmentsApi.update`, `billingApi.pay` on the client).
- **Notes**: Per-patient **`clinical_notes`** edited in the Activity tab; persisted with `PUT /api/v1/patients/{id}` (`patientsApi.update`). Organization-linked read-only doctors see notes without edit.
- **Deep links**: From the patient timeline, “Open in schedule” goes to `/doctor/appointments#appt-{id}`; the matching appointment card is **highlighted** (primary tint + ring) on `DoctorAppointmentsPage`.
- **Quick actions** from the header: **Book appointment** and **Create bill** navigate with router state so the **schedule** / **new bill** flows open with the **patient pre-selected** where applicable.

**E2E (Playwright):** `backend/frontend/e2e/doctor-patient-quick-actions.spec.ts` covers login → Book appointment → schedule slot → booking modal with patient prefill → Create bill → new bill modal with the patient’s visit. `doctor-patient-detail.spec.ts` also asserts hash navigation and the **ring** highlight on the target appointment.

### Admin creates doctor with login (backend)

Hospital **admin** and **super_admin** can create a doctor that includes a new login, tenant scoping, and optional **idempotent** retries:

- **Endpoint:** `POST /api/v1/doctors` with `DoctorCreate` including `name`, `specialization`, `experience_years`, `account_email`, `account_password`.
- **Access:** `admin` and `super_admin` only; **doctors** cannot create other doctors.
- **Tenancy:** `tenant_id` is resolved from `get_current_tenant_id`; if missing, **super_admin** may pass query `?tenant_id=`; otherwise a validation error is returned.
- **Transaction:** One commit in the service layer: `User` (role `doctor`, `force_password_reset=True`), `user_tenant` (primary, role `doctor`), `Doctor` profile; duplicate email handling aligns with CITEXT / uniqueness; optional **`Idempotency-Key`** header with stored request hash (same pattern as tenants/appointments).
- **Migrations:** `doctor_creation_idempotency` table (Alembic revision `a1b2c3d4e5f6` and later).
- **Tests:** `backend/tests/test_doctors.py` (API + idempotency; optional `@pytest.mark.postgres` concurrency test when `PYTEST_DATABASE_URL` points at PostgreSQL).

## Invariant System & Integrity Architecture

### Core Invariant Validation
**Location:** `@/backend/app/services/appointment_invariants.py`

The system enforces hard invariants linking appointments to their assigned doctors before any write operations that touch billing, inventory, or completion:

```python
def validate_appointment_invariants(appointment: Appointment, doctor: Doctor) -> None:
    # Enforces tenant and identity alignment
    if appointment.doctor_id != doctor.id:
        raise ValidationError("Appointment does not match the provided doctor identity")
    # Validates tenant consistency
    if appointment.tenant_id != doctor.tenant_id:
        raise ValidationError("Cross-tenant appointment detected")
```

**Usage:** Called before all critical operations (completion, billing, inventory usage).

### AppointmentInvariantGuard
**Location:** `@/backend/app/services/appointment_invariant_enforcement.py`

Single entry point for post-commit/load validation on appointment mutations:

- **`finalize(db, appointment_id)`** - Revalidates invariants after mutations and returns the reloaded appointment
- **`revalidate_appointment_invariants(appointment, db)`** - Pre-write validation before mutations
- **Mandatory** for all appointment mutation APIs to prevent bypass

**Pattern:** All appointment mutations end with `return AppointmentInvariantGuard.finalize(db, appointment_id)`

### Integrity Scan Service
**Location:** `@/backend/app/services/integrity_scan_service.py`

Read-only corruption detection and historical audit system:

**Features:**
- **Cross-tenant integrity scans** (super_admin only) with authorization checks
- **Per-tenant scans** for tenant administrators  
- **Thread-safe caching** with configurable TTL (default 5 minutes)
- **Structured issue reporting** with severity levels (critical, warning, info)
- **Comprehensive checks:** appointment-doctor alignment, billing consistency, inventory movements

**API Usage:**
- `GET /api/v1/admin/integrity-scan` - Trigger scan (admin/super_admin)
- Supports `?all_tenants=true` for super_admin cross-tenant scans
- Returns detailed integrity issues with timestamps and affected resources

### Idempotent Completion Enforcement
**Location:** `@/backend/app/services/appointment_service.py:469-510`

Appointment completion uses strict idempotency to prevent duplicate completions:

1. **Request body hashing** for content validation
2. **`appointment_completion_idempotency` table** for tracking
3. **Conflict detection** when same key used with different payload  
4. **Cached result return** for identical replays
5. **Invariant guard finalization** after successful completion

### Structured Audit Logging
All critical operations generate structured audit events with:
- **Tenant context** for cross-tenant visibility
- **User identity** and role information  
- **Resource IDs** and operation types
- **Timestamps** and outcome status
- **Security violation logging** for RBAC failures

**Example:**
```python
log_structured_audit_event(
    event="appointment_completed",
    tenant_id=appointment.tenant_id,
    user_id=current_user.id,
    resource_id=appointment.id,
    details={"doctor_id": doctor.id, "completion_method": "api"}
)
```

## Architecture Patterns

### Backend
- **Layering**: API routers → **services** (business rules, transactions, idempotency) → **CRUD** / ORM; Pydantic **schemas** for I/O
- **Dependency injection**: `Depends` for DB sessions and current user/tenant
- **Multi-Tenancy: Resource Ownership is Authoritative

- **`UserTenant`**, **`get_current_tenant_id`**: JWT/header tenant hints are secondary; **resource ownership (`resource.tenant_id`) is authoritative**
- **Tenant resolution authority**: For mutations, `align_operation_tenant_with_resource()` enforces that the resource's tenant_id matches the operation scope. Cross-tenant mismatches are rejected + audited.
- **Trust boundaries**: Frontend tenant IDs, role names, and JWT claims are NOT trusted alone. Resource ownership + invariants are the source of truth.

### Frontend
- **Container/Presentational pattern**: Pages fetch data, components render UI
- **Custom Hooks**: Data fetching and state logic encapsulated in hooks
- **Handler Pattern**: API calls abstracted in handlers folder
- **Tailwind + CSS**: Tailwind for layout/utilities, custom CSS for components

### Capability-based Authorization Refactor (May 2026)

**Problem:** Appointment completion authorization was incorrectly role-dependent. Admin users with Doctor records could not complete appointments for their assigned patients because the code checked `current_user.role == doctor` instead of verifying the user had a valid Doctor record linked to the appointment.

**Solution:** Refactored `authorize_appointment_access` and `mark_appointment_completed` to use **capability-based authorization**:

- **A user can complete an appointment IF:**
  1. A Doctor record exists where `Doctor.user_id == current_user.id`
  2. AND `doctor.id == appointment.doctor_id` (they are the assigned doctor)
  3. AND `doctor.tenant_id == appointment.tenant_id` (tenant matches)

**Key Changes:**
- `@/backend/app/services/appointment_service.py:607-633` — `_assert_doctor_assigned_to_appointment` now checks Doctor record existence + assignment match, independent of user role
- `@/backend/app/services/appointment_service.py:636-716` — `authorize_appointment_access` moved `require_assigned_doctor` check **before** role-based branching so ALL users with Doctor records (admin, staff, doctor) are evaluated the same way
- Removed hardcoded `if current_user.role != UserRole.doctor: raise ForbiddenError` from `mark_appointment_completed`

**Why This Matters:**
- Enables "admin-doctor" hybrid users (users with admin role who also have Doctor profiles)
- Future-proof for multi-tenant doctors, assistants, nurses with Doctor records
- Strict invariant: Only the assigned doctor can complete their appointments (enforced via Doctor record linkage, not role name)

## Architecture Pitfalls & Gotchas (For AI Context)

### 1. Multi-Tenancy is Everywhere
**Pattern:** Nearly every query must filter by `tenant_id`. The `get_current_tenant_id()` dependency resolver extracts tenant from JWT claims.

**Gotchas:**
- Some entities (like `Patient`) have **optional** `tenant_id` — patients can exist without a tenant until their first booking
- `Doctor` has `tenant_id` that is **mandatory** after creation
- `User` is global; `UserTenant` links users to tenants with roles per-tenant

**Critical Files:**
- `@/backend/app/core/tenant_context.py` — Tenant resolution; `align_operation_tenant_with_resource()` enforces resource ownership authority
- `@/backend/app/services/security_audit.py` — `assert_authorized()` validates cross-tenant access; resource tenant must match request scope

#### Superadmin Resource Scoping

`super_admin` is NOT unrestricted root access. The architecture enforces resource-level tenant scoping:

- **For mutations**: `align_operation_tenant_with_resource()` checks if `resource_tenant_id != request_tenant_id` → raises `ForbiddenError`
- **Cross-tenant blocked**: Attempts are logged with `[AUDIT] cross_tenant_blocked` and metric `cross_tenant_blocked_total` incremented
- **Result**: super_admin can operate across tenants, but ONLY when the resource belongs to the requested tenant scope. They cannot access resources in Tenant A while claiming to be in Tenant B.

### 2. Authorization is Role + Capability Hybrid
**Two authorization patterns coexist:**

| Pattern | Use Case | Files |
|---------|----------|-------|
| **Role-based** | Super admin bypass, patient self-access | `authorize_appointment_access` super_admin/patient branches |
| **Capability-based** | Doctor actions (appointment completion) | `_assert_doctor_assigned_to_appointment` |

**Rule of Thumb:**
- If the action requires being a "doctor" (completing visits, accessing patient records as provider), use **capability-based** via `get_current_doctor()` + Doctor record verification
- If the action is administrative (creating doctors, viewing all appointments), use **role-based**

**Critical Invariant:** Users act as doctors through `Doctor.user_id` linkage, NOT role names alone. An admin with a Doctor record can complete appointments.

### 3. Idempotency is Required for Mutations
**Any CREATE operation should support `Idempotency-Key` header:**
- Doctor creation (`POST /api/v1/doctors`)
- Appointment creation (`POST /api/v1/appointments`)
- Tenant creation
- Appointment completion (`POST /api/v1/appointments/{id}/mark-completed`)

**Pattern:**
1. Hash the request body
2. Check `*_idempotency` table for existing key
3. If exists with same hash → return cached result (idempotent replay)
4. If exists with different hash → `ConflictError`
5. Store key+hash+result after successful operation

**Critical Files:**
- `@/backend/app/services/appointment_service.py:469-510` — Completion idempotency
- `@/backend/app/crud/crud_appointment.py` — Idempotency table queries

### 4. Doctor-User Linkage is the Identity Anchor
**Key Concept:** Doctors are NOT users. A Doctor row represents a provider profile within a tenant.

- `Doctor.user_id` → links to User account (optional, but required for login)
- `Doctor.id` → the provider identity used in appointments, billing, etc.
- `User.role` → `admin`, `doctor`, `staff`, `patient`, `super_admin`

**Resolution:** `doctor_service.get_current_doctor(db, current_user)` looks up Doctor by `user_id`, NOT by `user.role`. This is critical for capability-based auth.

### 5. Frontend is Embedded in Backend
**Structure:**
```
backend/
├── app/           # FastAPI
└── frontend/      # React + Vite (NOT a separate repo)
    ├── src/
    └── dist/      # Built files served by FastAPI in production
```

**Development:**
- Frontend runs on `localhost:5173` (Vite dev server)
- Backend runs on `localhost:8000`
- CORS allows `localhost:5173` in dev
- In production, FastAPI serves the built frontend from `backend/frontend/dist`

### 6. Appointment State Machine & Invariant Enforcement
**Status transitions are enforced:**
- `scheduled` → `completed` (via `mark_appointment_completed`)
- `scheduled` → `cancelled`
- `completed` → anything (blocked by `AppointmentInvariantGuard`)

**Critical Invariant Checks:**
- **Doctor-Appointment Alignment:** `appointment.doctor_id` must match `doctor.id`
- **Tenant Consistency:** `appointment.tenant_id` must match `doctor.tenant_id`
- **Cross-tenant Prevention:** No appointment can span multiple tenants
- **Identity Verification:** Doctor actions require valid `Doctor.user_id` linkage

**Side Effects on Completion:**
1. Inventory deduction (if `data.items` provided)
2. Billing generation (if `data.generate_bill=True`) — billing inherits `appointment.tenant_id`
3. Slot cache invalidation
4. Idempotency record creation
5. **Invariant guard finalization** (mandatory post-commit validation)

**Appointment as Tenant Anchor:**
- `appointment.tenant_id` is **mandatory** (non-nullable) and serves as the authoritative tenant context
- Billing inherits appointment tenant: integrity scan validates `bill.tenant_id == appointment.tenant_id`
- Inventory usage must match appointment tenant: movements are validated against appointment invariants
- Patients do NOT own tenant context: `patient.tenant_id` is always NULL; patient access is scoped via appointments

### Trust Boundaries

Explicit trust boundaries prevent AI agents or developers from reintroducing multi-tenant bugs:

**Rule 1: Frontend tenant IDs are NOT trusted**
- Headers like `X-Tenant-ID` are hints only; must be validated against `user_tenant` membership
- Resource ownership (`appointment.tenant_id`, `billing.tenant_id`) is authoritative

**Rule 2: Role names alone are NOT trusted**
- Capability-based authorization via `Doctor.user_id` linkage
- `current_user.role == "doctor"` checks are forbidden; use `get_current_doctor()` + record verification

**Rule 3: JWT/header tenant hints are NOT authoritative alone**
- Validated against `user_tenant` membership AND resource ownership
- `align_operation_tenant_with_resource()` enforces resource tenant matches request scope

**Rule 4: Resource ownership + invariants are authoritative**
- `assert_authorized()` enforces `resource_tenant_id == tenant_id`
- Appointment invariants enforce doctor/tenant alignment via `validate_appointment_invariants()`

**Enforcement Points:**
- `@/backend/app/core/tenant_context.py:181-209` — `align_operation_tenant_with_resource()`
- `@/backend/app/services/security_audit.py:31-45` — `enforce_tenant_match()`
- `@/backend/app/services/appointment_invariants.py` — `validate_appointment_invariants()`

### 7. Common Bugs to Avoid

**Don't:**
```python
# ❌ Role-based check for doctor actions
if current_user.role != UserRole.doctor:
    raise ForbiddenError("Only doctors can do this")
```

**Do:**
```python
# ✅ Capability-based check
doc = doctor_service.get_current_doctor(db, current_user)
if doc.id != appointment.doctor_id:
    raise ForbiddenError("Only the assigned doctor can complete this appointment")
```

**Don't:**
```python
# ❌ Forget tenant check
doctor = crud_doctor.get_doctor(db, doctor_id)  # Could be from any tenant!
```

**Do:**
```python
# ✅ Always scope by tenant
doctor = crud_doctor.get_doctor(db, doctor_id, tenant_id=current_tenant_id)
# OR verify after fetch:
if doctor.tenant_id != appointment.tenant_id:
    raise ForbiddenError("Cross-tenant access not allowed")
```

### 8. Testing Strategy
**Backend:** pytest with optional PostgreSQL tests via `@pytest.mark.postgres`
- Run with SQLite: `pytest backend/tests/`
- Run with PostgreSQL: `PYTEST_DATABASE_URL=postgresql://... pytest backend/tests/ -m postgres`

**Frontend:** Playwright E2E
- `cd backend/frontend && npx playwright test`
- Tests cover: auth, patient management, doctor workflows, appointment completion

### 9. Integrity Scanning & Corruption Detection
**Use the built-in integrity scan system:**
- **Cross-tenant scans:** Super admin can scan all tenants with `?all_tenants=true`
- **Per-tenant scans:** Tenant admins scan their own organization
- **Cached results:** 5-minute TTL prevents excessive database load
- **Structured issues:** CRITICAL and WARNING severity levels (INFO does not exist)

**When to run:**
- Pre-deployment validation
- Suspicion of data corruption  
- Regular maintenance (cron job)

### 10. Database Migrations
**Always use Alembic:**
```bash
cd backend
alembic revision --autogenerate -m "description"
alembic upgrade head
```

**Critical:** Some migrations are **NOT** autogeneratable (e.g., adding CITEXT columns, complex constraints). Check recent migrations for patterns.

- Add router in `@/backend/app/api/v1/endpoints/`
- Register in `@/backend/app/api/v1/api.py`
- Follow existing patterns for: auth, idempotency, tenant scoping, error handling
