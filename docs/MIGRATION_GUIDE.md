## 4. Creating New Migrations

### Step 1: Generate

```bash
cd backend
alembic revision --autogenerate -m "z9_description_of_change"
```

### Step 2: Review

Always review the autogenerated migration before applying:

```bash
# Check what was generated
cat alembic/versions/z9_*.py
```

### Step 3: Verify

```bash
# Dry run (offline mode)
alembic upgrade head --sql

# Check current state
alembic current
```

### Step 4: Apply

```bash
alembic upgrade head
```

### Step 5: Verify

```bash
alembic current  # Should show your new revision
```

## 5. Migration Safety Rules

### Additive Only

**NEVER** drop columns or tables in migrations. All changes must be additive:

```python
# ✅ SAFE: Add nullable column
op.add_column('appointments', sa.Column('clinical_notes', sa.Text(), nullable=True))

# ❌ DANGEROUS: Drop column
op.drop_column('appointments', 'old_field')

# ❌ DANGEROUS: Drop table
op.drop_table('obsolete_table')
```

### Nullable New Columns

New columns should be nullable for backward compatibility:

```python
# ✅ SAFE: Nullable new column
sa.Column('new_field', sa.String(), nullable=True)

# ⚠️ RISKY: Non-nullable new column (requires server_default)
sa.Column('new_field', sa.String(), nullable=False, server_default='default_value')
```

### Idempotent PostgreSQL Helpers

Use `migration_helpers.py` for PostgreSQL-specific operations:

```python
from migration_helpers import (
    create_citext_extension,
    safe_create_enum,
    safe_add_column,
)

def upgrade():
    safe_create_enum(op, 'appointmentstatus', ['scheduled', 'checked_in', 'in_consultation', 'completed', 'cancelled'])
    safe_add_column(op, 'appointments', sa.Column('new_field', sa.String(), nullable=True))
```

### SQLite Compatibility

For test compatibility, avoid PostgreSQL-only types in migrations:

| PostgreSQL Type | SQLite Alternative |
|-----------------|-------------------|
| `ARRAY` | Use JSON string or separate table |
| `JSONB` | Use JSON string |
| `CITEXT` | Use `String` + unique constraint |
| `ENUM` | Use `String` + check constraint |
| `DISTINCT ON` | Use subquery |

## 6. Production Deployment Safety

### Pre-Deployment Checklist

- [ ] Migration is additive only
- [ ] Migration reviewed and tested on staging
- [ ] Full database backup taken
- [ ] `alembic upgrade head --sql` dry-run passes
- [ ] No pending migrations in any environment
- [ ] Staging DB is in sync with production

### Deployment Command

```bash
# Production start command (Render)
bash -c "set -euo pipefail; alembic upgrade head; exec uvicorn app.main:app --host 0.0.0.0 --port ${PORT:-10000}"
```

### Rollback

```bash
# Rollback one migration
alembic downgrade -1

# Rollback to specific revision
alembic downgrade <revision_id>
```

**Note:** Rollback is only safe for additive migrations. If a migration added columns, downgrade drops them.

## 7. Merge Migration Handling

### When Merge Migrations Are Needed

Merge migrations are required when two branches of development create independent revisions that both depend on the same parent:

```
parent_revision
  ├── branch_a_revision
  └── branch_b_revision
       └── merge_revision (depends on both)
```

### Creating a Merge Migration

```bash
alembic merge -m "merge branch_a and branch_b" branch_a_revision branch_b_revision
```

### Current Merge Points

The graph has two merge points:
1. `p1q2r3s4t6` — Merge of two branches

## 8. Baseline Strategy

### Current Approach

The migration graph is healthy and maintainable at 49 files. A baseline snapshot is **not needed now** but is documented for future use.

### When to Baseline

Consider a baseline snapshot when:
1. Migration count exceeds 100 files
2. New team members consistently struggle with the migration history
3. Fresh database provisioning takes more than 5 minutes
4. A new major version (v2.0) is being released

### Baseline Approach

See `backend/MIGRATION_CLEANUP_STRATEGY.md` for the documented plan:

1. Create a baseline migration with `down_revision = None`
2. Archive old revisions to `alembic/archive/`
3. Use baseline only for **new database provisioning**
4. Production keeps its existing `alembic_version` entry

**Do NOT stamp or modify the production `alembic_version` table.**

## 9. Common Issues

### "Can't locate revision" Error

**Root cause:** Migration file not present in deployed artifact.

**Fix:**
1. Ensure `alembic/versions/` is not in `.gitignore`
2. Verify deployment includes all migration files
3. Run `alembic upgrade head` as part of CI/CD, not just at startup

### Filename vs Revision Mismatch

**Known issue:** `z1_full_tenant_cleanup_hardening.py` has internal `revision = "z1_full_tenant_cleanup"`.

**Impact:** Cosmetic only. Alembic uses the internal `revision` field, not the filename. **Do not fix** — changing it would break production compatibility.

### Offline Mode Failures

Some migrations use `sa.inspect()` which fails in offline mode (`--sql` flag).

**Workaround:** Skip offline mode for these migrations, or test with a real database.

## 11. Environment Configuration

### `alembic.ini`

```ini
[alembic]
script_location = alembic
sqlalchemy.url = driver://user:pass@localhost/dbname
```

### `env.py`

```python
from app.core.config import settings
from app.core.database import Base
import app.models  # Ensure all models are loaded

target_metadata = Base.metadata

def run_migrations_online():
    connectable = create_engine(settings.DATABASE_URL)
    with connectable.connect() as connection:
        context.configure(
            connection=connection,
            target_metadata=target_metadata,
        )
        with context.begin_transaction():
            context.run_migrations()
```

## 12. Testing Migrations

### Staging Verification

1. Provision a fresh staging database
2. Run `alembic upgrade head`
3. Verify all tables, enums, indexes, and constraints exist
4. Run the application test suite
5. Run integrity scans

### Test Database

Tests use SQLite in-memory by default. PostgreSQL-only tests use `@pytest.mark.postgres`:

```bash
# Default (SQLite)
pytest backend/tests/

# PostgreSQL
$env:PYTEST_DATABASE_URL="postgresql://..."; pytest -m postgres
```
